# I/O 스케줄링 프레임워크: 다중 애플리케이션을 위한 아두이노 하드웨어 추상화

## 한 줄 요약

라즈베리파이 위에서 여러 프로그램이 하나의 아두이노 하드웨어를 동시에 사용할 수 있게 해주는 **중간 관리자(데몬)**를 만들고, 요청을 처리하는 **스케줄링 정책을 교체**할 수 있게 설계한다.

---

## 1. 문제 정의

### 일반적인 상황

아두이노에 센서(온도, 조도 등)와 액추에이터(모터, LED 등)가 연결되어 있고, 라즈베리파이가 시리얼(UART) 통신으로 아두이노를 제어한다고 하자.

### 문제

라즈베리파이에서 **프로그램 하나**가 시리얼 포트를 열면, 다른 프로그램은 그 포트를 사용할 수 없다. 즉, 아두이노에 연결된 하드웨어를 **한 번에 하나의 프로그램만** 쓸 수 있다.

예를 들어:
- 프로그램 A: 온도 센서를 1초마다 읽고 싶다
- 프로그램 B: LED를 제어하고 싶다
- 프로그램 C: 모터를 돌리고 싶다

이 세 프로그램이 동시에 실행되면, 시리얼 포트를 누가 쓸 것인지 충돌이 발생한다.

### 현실에서의 비유

프린터가 하나인 사무실을 생각하면 된다. 여러 사람이 동시에 인쇄를 요청하면 프린터 스풀러(관리자)가 요청을 순서대로 정리해서 하나씩 처리한다. 이 프로젝트는 **아두이노 하드웨어에 대한 프린터 스풀러**를 만드는 것이다.

## 2. 해결 방법

### 전체 구조

```
[라즈베리파이]
┌──────────────────────────────────────────────────┐
│                                                  │
│   App A ──┐                                      │
│           │    ┌──────────────────────────┐       │
│   App B ──┼───→│   I/O 스케줄링 데몬      │       │      [아두이노]
│           │    │                          │       │    ┌────────────────┐
│   App C ──┘    │  ┌────────────────────┐  │  UART │    │ 온도 센서      │
│                │  │ 스케줄링 정책       │  │───────┼───→│ LED            │
│                │  │ (FIFO/우선순위/EDF) │  │       │    │ 모터           │
│                │  └────────────────────┘  │       │    │ ...            │
│                └──────────────────────────┘       │    └────────────────┘
│                                                  │
└──────────────────────────────────────────────────┘
```

### 핵심 구성 요소

#### (1) I/O 스케줄링 데몬 (라즈베리파이)

시스템이 부팅되면 백그라운드에서 실행되는 프로세스(데몬)다. 이 데몬이 **시리얼 포트를 독점**하고, 모든 앱의 요청을 대신 처리한다.

- 앱들로부터 I/O 요청을 받는다 (Unix 도메인 소켓 등을 통해)
- 요청을 내부 큐에 넣는다
- **스케줄링 정책**에 따라 어떤 요청을 먼저 처리할지 결정한다
- 아두이노에 명령을 보내고, 응답을 받아서 해당 앱에 돌려준다

#### (2) 스케줄링 정책 엔진

데몬 내부에서 요청의 처리 순서를 결정하는 모듈이다. **런타임에 정책을 교체**할 수 있도록 설계한다.

구현할 정책 예시:

| 정책 | 설명 | 적합한 상황 |
|------|------|------------|
| **FIFO** | 먼저 들어온 요청을 먼저 처리 | 공평성이 중요할 때 |
| **우선순위 기반** | 높은 우선순위의 요청을 먼저 처리 | 긴급한 제어 명령이 있을 때 |
| **SJF (Shortest Job First)** | 예상 소요 시간이 가장 짧은 요청을 먼저 처리 | 평균 대기 시간을 최소화할 때 |
| **SRTF (Shortest Remaining Time First)** | 현재 처리 중인 요청보다 더 짧은 요청이 들어오면 선점(preempt)하여 먼저 처리 | 짧은 요청의 응답성이 중요할 때 |
| **EDF (Earliest Deadline First)** | 마감 시간이 가장 가까운 요청을 먼저 처리 | 실시간 제약이 있을 때 |
| **라운드 로빈** | 앱들에게 순서대로 공평하게 시간을 배분 | 모든 앱이 균등하게 접근해야 할 때 |

> **SJF/SRTF가 가능한 이유**: 일반 OS에서는 프로세스 실행 시간을 예측할 수 없어 SJF/SRTF는 이론적으로만 최적인 정책으로 취급된다. 그러나 이 프레임워크에서는 아두이노 하드웨어 작업의 소요 시간이 **거의 결정적(deterministic)**이다. 센서 읽기나 액추에이터 제어는 매번 비슷한 시간이 걸리기 때문에, 실행 시간 예측이 현실적으로 가능하다.
>
> 소요 시간 추정 방식:
> - **정적 등록**: 디바이스 등록 시 예상 소요 시간을 명시 (예: 온도 센서 읽기 = 10ms)
> - **런타임 측정**: 실제 응답 시간을 누적하여 지수 이동 평균(exponential moving average)으로 보정. 초기값은 정적으로 제공하고, 사용할수록 정확해진다.

#### (3) 클라이언트 라이브러리

앱 개발자가 데몬과 통신하기 위해 사용하는 API다. 앱은 아두이노와 직접 통신하는 것처럼 간단하게 코드를 작성할 수 있다.

```c
// 앱 개발자가 사용하는 코드 (예시)
io_handle_t *handle = io_connect();              // 데몬에 연결
io_read(handle, SENSOR_TEMP, &value);            // 온도 읽기 요청
io_write(handle, ACTUATOR_LED, LED_ON);          // LED 켜기 요청
io_disconnect(handle);                           // 연결 종료
```

내부적으로는 Unix 소켓을 통해 데몬에 요청을 보내고 응답을 기다리는 구조지만, 앱 개발자는 이를 신경 쓸 필요가 없다.

#### (4) 아두이노 펌웨어

아두이노 측에서 라즈베리파이로부터 명령을 받아 실행하는 펌웨어다.

- 시리얼로 명령 수신 (예: `READ TEMP`, `WRITE LED 1`)
- 해당 센서를 읽거나 액추에이터를 제어
- 결과를 시리얼로 응답

## 3. 이 프로젝트에서 다루는 핵심 개념

| 개념 | 설명 |
|------|------|
| **하드웨어 추상화** | 앱이 특정 하드웨어의 세부사항(시리얼 프로토콜, 핀 번호 등)을 몰라도 사용할 수 있게 인터페이스를 제공 |
| **I/O 스케줄링** | 여러 요청이 동시에 들어올 때, 어떤 순서로 처리할지를 결정하는 정책. OS의 프로세스 스케줄링과 유사한 개념을 I/O 레벨에 적용 |
| **IPC (프로세스 간 통신)** | 앱과 데몬이 서로 데이터를 주고받는 메커니즘. Unix 도메인 소켓 사용 |
| **동시성 제어** | 여러 앱이 동시에 요청을 보낼 때 데이터 충돌이 나지 않도록 락, 큐 등을 사용하여 관리 |
| **디바이스 다중화** | 하나의 물리적 장치(아두이노)를 여러 논리적 사용자가 공유하는 것 |

## 4. 기대 결과물

1. **I/O 데몬 프로세스** — C로 작성, 라즈베리파이에서 실행
2. **클라이언트 라이브러리** — C API, 앱 개발자용
3. **아두이노 펌웨어** — 명령 처리 + 하드웨어 제어
4. **통신 프로토콜 명세** — 데몬↔앱, 라즈베리파이↔아두이노 간 메시지 포맷 정의
5. **벤치마크 결과** — 스케줄링 정책별 응답 시간, 처리량 비교 그래프
6. **샘플 애플리케이션 2~3개** — 프레임워크 위에서 동작하는 데모 앱
