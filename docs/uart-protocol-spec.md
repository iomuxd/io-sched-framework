# UART 통신 프로토콜 명세서

> **문서 버전**: v0.1 (초안)
> **대상 하드웨어**: Arduino Uno (ATmega328P) ↔ Raspberry Pi
> **통신 방식**: 동기식 요청-응답 (Request-Response)

---

## 1. 개요

본 프로토콜은 라즈베리파이의 I/O 스케줄링 데몬이 아두이노에 연결된 센서/액추에이터를
제어하기 위한 시리얼 통신 규약이다.

### 1.1 설계 목표

| 원칙 | 설명 |
|------|------|
| 단순성 | Uno의 제한된 리소스(2KB SRAM, 64B 시리얼 버퍼)에서 안정적으로 동작할 것 |
| 신뢰성 | 노이즈 환경(점퍼 와이어)에서도 데이터 무결성을 보장할 것 |
| 대칭성 | 요청/응답 프레임 구조를 통일하여 파싱 로직을 재사용할 수 있을 것 |
| 디버깅 용이성 | 오실로스코프, 로그만으로 문제를 추적할 수 있을 것 |

### 1.2 물리 계층 파라미터

```
  Baud Rate :  115200 bps
  Data Bits :  8
  Parity    :  None
  Stop Bits :  1
  표기      :  8N1
```

115200은 Uno에서 안정적으로 동작하는 최대 속도이다.
이보다 높은 속도(예: 250000)는 클럭 분주 오차로 인해 비트 에러율이 증가한다.

### 1.3 통신 모델

```
  ┌──────────────┐         요청           ┌──────────────┐
  │              │ ─────────────────────→ │              │
  │  Raspberry   │                        │   Arduino    │
  │     Pi       │ ←───────────────────── │     Uno      │
  │              │         응답           │              │
  └──────────────┘                        └──────────────┘
```

* 동기식: 요청 1개를 보내면, 반드시 응답 1개를 받은 뒤 다음 요청을 보낸다.
* 항상 라즈베리파이가 요청을 시작한다 (Master-Slave 구조).
* 아두이노는 자발적으로 데이터를 보내지 않는다.

## 2. 프레임 구조

### 2.1 요청 프레임 (Raspberry Pi → Arduino)

```
  바이트    0        1        2        3       4    5        6        7
  ┌────────┬────────┬────────┬────────┬─────────────┬────────┬────────┐
  │  SOF   │  SEQ   │  CMD   │  DEV   │   PAYLOAD   │  CRC8  │  EOF   │
  └────────┴────────┴────────┴────────┴─────────────┴────────┴────────┘
  │ 1 byte │ 1 byte │ 1 byte │ 1 byte │   2 bytes   │ 1 byte │ 1 byte │

  총 크기: 8 bytes (고정)
```

### 2.2 응답 프레임 (Arduino → Raspberry Pi)

```
  바이트    0        1        2        3       4    5        6        7
  ┌────────┬────────┬────────┬────────┬─────────────┬────────┬────────┐
  │  SOF   │  SEQ   │ STATUS │  DEV   │   PAYLOAD   │  CRC8  │  EOF   │
  └────────┴────────┴────────┴────────┴─────────────┴────────┴────────┘
  │ 1 byte │ 1 byte │ 1 byte │ 1 byte │   2 bytes   │ 1 byte │ 1 byte │

  총 크기: 8 bytes (고정)
```

### 2.3 요청/응답 프레임 비교

```
  요청:  │ SOF │ SEQ │  CMD   │ DEV │ PAYLOAD │ CRC8 │ EOF │
         ├─────┼─────┼────────┼─────┼─────────┼──────┼─────┤
  응답:  │ SOF │ SEQ │ STATUS │ DEV │ PAYLOAD │ CRC8 │ EOF │
                      ^^^^^^^^
                  유일한 차이점:
                  요청에서는 "무엇을 할지" (CMD)
                  응답에서는 "결과가 어떤지" (STATUS)
```

**구조를 통일한 이유**: 아두이노 펌웨어에서 응답을 만들 때,
수신한 요청 버퍼를 복사한 뒤 CMD → STATUS 교체, PAYLOAD 채우기,
CRC 재계산만 하면 된다. 별도의 응답 빌더가 필요 없다.

## 3. 필드 상세

### 3.1 SOF (Start of Frame)

```
  비트 패턴: 1 0 1 0 1 0 1 0
  값:       0xAA
  위치:     오프셋 0
```

**역할**: 프레임의 시작 경계를 표시한다.

UART는 연속된 바이트 스트림이므로 프레임 간 경계가 존재하지 않는다.
노이즈나 바이트 유실로 동기화가 깨지면, 수신 측은 0xAA가 나타날 때까지
바이트를 버리고 해당 지점부터 프레임 파싱을 재시도한다.

**0xAA를 선택한 이유**: 비트 패턴 `10101010`은 오실로스코프에서
균일한 구형파로 표시되어 육안 식별이 용이할 수 있다고 한다.


### 3.2 SEQ (Sequence Number)

```
  범위: 0x00 ~ 0xFF (0 ~ 255, 이후 0으로 순환)
  위치: 오프셋 1
```

**역할**: 요청-응답 쌍을 매칭한다.

동기식 통신에서는 응답이 항상 직전 요청에 대한 것이므로 이론적으로는
불필요하다. 그러나 다음 상황에서 필수적이다:

```
  정상 시나리오:
    Pi  ──[SEQ=1, READ]──→  Uno
    Pi  ←──[SEQ=1, OK]────  Uno     ✓ 매칭 확인

  타임아웃 + 재전송 시나리오:
    Pi  ──[SEQ=1, READ]──→  Uno
    Pi  (타임아웃, 응답 없음)
    Pi  ──[SEQ=2, READ]──→  Uno     재전송 (새 SEQ)
    Pi  ←──[SEQ=1, OK]────  Uno     늦게 도착한 이전 응답 → SEQ 불일치 → 폐기
    Pi  ←──[SEQ=2, OK]────  Uno     ✓ 올바른 응답
```

응답 프레임에서는 요청의 SEQ를 **그대로 에코**한다.
별도 번호 체계를 운영하지 않음으로써 양쪽의 상태 관리를 최소화한다.


### 3.3 CMD (Command) — 요청 프레임 전용

```
  위치: 오프셋 2 (요청 프레임에서만 사용)
```

| 코드   | 이름    | 설명                    | PAYLOAD 사용       |
|--------|---------|------------------------|--------------------|
| `0x01` | `READ`  | 센서 값 읽기            | 요청: 미사용 (0x0000) |
| `0x02` | `WRITE` | 액추에이터 제어          | 요청: 제어 값       |
| `0x03` | `PING`  | 연결 상태 확인           | 요청: 미사용        |
| `0x04` | `INFO`  | 디바이스 정보 조회       | 요청: 미사용        |

**CMD를 별도 필드로 분리한 이유**: PAYLOAD에 명령 종류를 인코딩하면
아두이노에서 페이로드를 먼저 파싱해야 동작을 판단할 수 있다.
CMD를 독립 필드로 두면 `switch(frame.cmd)` 한 줄로 분기 가능하다.


### 3.4 STATUS — 응답 프레임 전용

```
  위치: 오프셋 2 (응답 프레임에서만 사용, CMD와 같은 위치)
```

| 코드   | 이름              | 설명                              |
|--------|-------------------|-----------------------------------|
| `0x00` | `OK`              | 요청 성공                          |
| `0x01` | `ERR_UNKNOWN_DEV` | 존재하지 않는 디바이스 ID           |
| `0x02` | `ERR_UNKNOWN_CMD` | 지원하지 않는 명령 코드             |
| `0x03` | `ERR_CRC`         | 수신 프레임의 CRC 불일치            |
| `0x04` | `ERR_DEV_TIMEOUT` | 디바이스(센서/액추에이터) 응답 없음  |

**CMD와 같은 오프셋을 공유하는 이유**: 응답에는 명령 코드를 재전송할
필요가 없다. 동기식이므로 어떤 명령에 대한 응답인지는 SEQ로 이미
추적 가능하다. 같은 위치를 재활용하여 프레임 크기를 늘리지 않는다.


### 3.5 DEV (Device ID)

```
  범위: 0x00 ~ 0xFF
  위치: 오프셋 3
```

명령 대상 디바이스를 지정한다.
응답에서는 요청의 DEV를 그대로 에코한다.

예약된 ID:

| ID     | 용도                    |
|--------|------------------------|
| `0x00` | 시스템 전체 (PING 등)   |
| `0x01~` | 사용자 등록 디바이스    |

**디바이스 등록 예시** (펌웨어에서 정적 정의):

```
  DEV 0x01 = LED            (D9 핀, 디지털 출력,        ~1ms)
  DEV 0x02 = DHT11 온습도   (D2 핀, 원와이어 프로토콜,  ~20ms)
  DEV 0x03 = 서보 SG90      (D10 핀, PWM 제어,          ~수백ms)
  DEV 0x04 = HC-SR04 초음파  (TRIG=D7, ECHO=D8,         ~30ms)
```


### 3.6 PAYLOAD

```
  크기: 2 bytes (리틀 엔디안)
  위치: 오프셋 4~5
```

```
  바이트 순서 (리틀 엔디안):
    오프셋 4 = 하위 바이트 (LSB)
    오프셋 5 = 상위 바이트 (MSB)

  예: 값 1023 (0x03FF)
    오프셋 4 = 0xFF
    오프셋 5 = 0x03
```

**2바이트로 고정한 이유**:

```
  Uno ADC 해상도:  10비트 → 0 ~ 1023       → 2바이트로 충분
  PWM 출력 범위:    8비트 → 0 ~ 255        → 1바이트로도 충분
  서보 각도:                 0 ~ 180        → 1바이트로도 충분
  디지털 출력:               0 또는 1       → 1바이트로도 충분
```

어떤 디바이스든 2바이트면 값 표현에 충분하다.
가변 길이로 하면 프레임 크기를 동적으로 파싱해야 하므로
Uno의 제한된 SRAM에서 구현 복잡도가 불필요하게 증가한다.

**명령별 PAYLOAD 사용**:

| CMD | 요청 PAYLOAD | 응답 PAYLOAD |
|-----|-------------|-------------|
| READ | 0x0000 (미사용) | 센서 값 (0~1023 등) |
| WRITE | 제어 값 | 0x0000 또는 에코 |
| PING | 0x0000 | 0x0000 |
| INFO | 0x0000 | 디바이스 타입 코드 |


### 3.7 CRC8 (Cyclic Redundancy Check, 8-bit)

```
  다항식:    CRC-8-CCITT (x⁸ + x² + x + 1), 0x07
  초기값:    0x00
  계산 범위: 오프셋 1 ~ 5 (SEQ, CMD/STATUS, DEV, PAYLOAD)
  위치:      오프셋 6
```

**SOF/EOF를 CRC 계산에서 제외하는 이유**: SOF와 EOF는 고정 값(0xAA, 0x55)
이므로 CRC에 포함해도 검출 능력이 향상되지 않는다. 가변적인 데이터 영역만
보호하는 것이 효율적이다.

**체크섬 대신 CRC를 사용하는 이유**: 단순 합산 체크섬은 바이트 순서가
뒤바뀌는 에러를 검출하지 못한다. CRC-8은 Uno에서 연산 부하가 거의 없으면서
(시프트 연산 기반) 1~2비트 에러를 확실하게 검출한다.

### 3.8 EOF (End of Frame)

```
  비트 패턴: 0 1 0 1 0 1 0 1
  값:       0x55
  위치:     오프셋 7
```

**역할**: 프레임의 끝 경계를 표시한다.

고정 길이 프레임이므로 SOF만으로도 프레임 경계를 잡을 수 있지만,
EOF로 이중 검증하면 동기화 복구의 신뢰도가 높아진다.

```
  SOF만 사용할 경우:
    ... [노이즈] [0xAA] [쓰레기 6바이트] [0xAA] [정상 프레임] ...
                 ^^^^^^^^^^^^^^^^^^^^^^^^
                 잘못된 프레임으로 오인 가능

  SOF + EOF 사용할 경우:
    ... [노이즈] [0xAA] [쓰레기 6바이트] [0xAA] [정상 프레임] [0x55] ...
                 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                 EOF 위치 검증 실패 → 폐기, 다음 SOF 탐색
```

**0x55를 선택한 이유**: SOF(`0xAA` = `10101010`)와 정확히 비트 반전된
패턴(`01010101`)이므로 오실로스코프에서 시작과 끝을 즉시 구분할 수 있다고 한다.

## 4. 통신 예시

### 4.1 온도 센서 읽기 (성공)

```
  요청: DEV=0x01(온도센서)에 READ 명령
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ 0xAA │ 0x01 │ 0x01 │ 0x01 │ 0x00 │ 0x00 │ 0xXX │ 0x55 │
  │ SOF  │ SEQ  │ READ │ DEV  │  PAYLOAD(0) │ CRC8 │ EOF  │

  응답: 온도 값 = 512 (0x0200)
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ 0xAA │ 0x01 │ 0x00 │ 0x01 │ 0x00 │ 0x02 │ 0xXX │ 0x55 │
  │ SOF  │ SEQ  │  OK  │ DEV  │ PAYLOAD(512)│ CRC8 │ EOF  │
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

## 5. 에러 처리

### 5.1 타임아웃

```
  디바이스별 타임아웃 (하드코딩):

  │ DEV  │ 디바이스       │ 예상 응답 │ 타임아웃 │
  │ 0x01 │ LED            │    ~1ms   │   50ms   │
  │ 0x02 │ DHT11 온습도   │   ~20ms   │  200ms   │
  │ 0x03 │ 서보 SG90      │ ~수백ms   │ 1000ms   │
  │ 0x04 │ HC-SR04 초음파 │   ~30ms   │  200ms   │

  ┌───── Pi ─────┐                    ┌─── Uno ────┐
  │              │ ──[SEQ=N]───────→  │            │
  │              │                    │ (처리 중   │
  │ 타임아웃 경과│                    │ 또는 고장) │
  │              │ ──[SEQ=N+1]─────→  │            │  ← 재전송 (새 SEQ)
  │              │ ←──[SEQ=N+1]────── │            │
  └──────────────┘                    └────────────┘

  • 최대 재시도 횟수: 3회
  • 3회 실패 시 해당 요청을 에러로 상위 레이어에 반환

  한계: 타임아웃 값은 DEV ID 기준으로 데몬에 하드코딩된다.
  새 디바이스 추가 시 코드 수정이 필요하며, 런타임 설정은
  이번 프로젝트의 관심사가 아니므로 지원하지 않는다.
```

### 5.2 CRC 불일치

```
  수신 측에서 CRC 검증 실패 시:

  [아두이노가 수신한 경우]
    → STATUS = ERR_CRC (0x03) 응답 전송
    → 요청의 다른 필드는 신뢰할 수 없으므로 SEQ=0x00, DEV=0x00으로 응답

  [라즈베리파이가 수신한 경우]
    → 해당 응답 폐기
    → 타임아웃과 동일하게 재전송 처리
```

### 5.3 프레임 동기화 복구

```
  정상 수신 상태 머신:

       바이트 수신
           │
           ▼
  ┌─────────────────┐    != 0xAA      ┌────────┐
  │  SOF 대기       │ ──────────────→ │  폐기  │
  │  (IDLE)         │ ←────────────── └────────┘
  └────────┬────────┘
           │ == 0xAA
           ▼
  ┌─────────────────┐
  │  나머지 7바이트 │
  │  수신 중        │
  └────────┬────────┘
           │ 7바이트 수신 완료
           ▼
  ┌─────────────────┐    != 0x55      ┌──────────────────┐
  │  EOF 검증       │ ──────────────→ │  프레임 폐기     │
  └────────┬────────┘                 │  SOF 대기로 복귀 │
           │ == 0x55                  └──────────────────┘
           ▼
  ┌─────────────────┐    불일치       ┌──────────────────┐
  │  CRC 검증       │ ──────────────→ │ 프레임 폐기      │
  └────────┬────────┘                 │ (에러 응답 전송) │
           │ 일치                     └──────────────────┘
           ▼
  ┌─────────────────┐
  │  프레임 처리    │
  └─────────────────┘
```

## 6. 설계 결정 요약

| 결정 사항 | 선택 | 근거 |
|----------|------|------|
| 통신 방식 | 동기식 | Uno 64B 버퍼에서 비동기 파이프라이닝은 오버플로우 위험. 프로젝트 핵심은 스케줄링이지 UART 최적화가 아님 |
| 프레임 크기 | 8바이트 고정 | 필드 합산 결과. 고정 길이라 파싱 단순, Uno SRAM 사용 예측 가능 |
| PAYLOAD 크기 | 2바이트 고정 | ADC 10비트(0~1023) 커버. 가변 길이 시 파싱 복잡도 불필요하게 증가 |
| 바이트 순서 | 리틀 엔디안 | ATmega328P, ARM(RPi) 모두 리틀 엔디안 아키텍처 |
| 무결성 검증 | CRC-8-CCITT | 단순 체크섬 대비 에러 검출 우수. Uno에서 연산 부하 무시 가능 |
| 프레임 구조 대칭 | 요청 ≈ 응답 | 파싱 로직 재사용 가능. 응답 생성 시 버퍼 복사 후 최소 필드만 수정 |
