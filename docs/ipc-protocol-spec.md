# 앱 ↔ 데몬 IPC 프로토콜 명세서

> **문서 버전**: v0.2
> **통신 채널**: Unix 도메인 소켓 (SOCK_STREAM)
> **대상 환경**: Raspberry Pi OS (Linux)

---

## 1. 개요

본 프로토콜은 클라이언트 앱과 I/O 스케줄링 데몬 간의 통신 규약이다.
앱은 이 프로토콜을 통해 아두이노에 연결된 센서/액추에이터에 대한
I/O 요청을 데몬에 제출하고, 데몬은 스케줄링 정책에 따라 요청을 처리한다.

### 1.1 설계 목표

| 목표 | 설명 |
|------|------|
| 비동기 제출 | 앱이 응답을 기다리지 않고 여러 요청을 연속 제출할 수 있을 것 |
| 스케줄링 지원 | 요청마다 우선순위, 데드라인 등 스케줄링 힌트를 첨부할 수 있을 것 |
| 다중 클라이언트 | 여러 앱이 동시에 데몬에 접속하여 요청을 보낼 수 있을 것 |
| 단순한 API | 클라이언트 라이브러리를 통해 소켓 세부사항을 숨길 것 |

### 1.2 UART 프로토콜과의 차이

| 항목 | UART (데몬 ↔ 아두이노) | IPC (앱 ↔ 데몬) |
|------|------------------------|------------------|
| 전송 매체 | 시리얼 (물리 와이어) | Unix 도메인 소켓 (커널 내부) |
| 신뢰성 | 노이즈 취약 → CRC, SOF/EOF 필요 | 커널이 무결성 보장 → 불필요 |
| 프레임 크기 | 8B 고정 | 가변 길이 (LEN 필드) |
| 통신 방식 | 동기식 (요청-응답 1:1) | 비동기식 (요청 다수 제출, 응답 개별 수신) |
| 식별자 크기 | SEQ 1B (0~255) | REQ_ID 4B (다중 앱 충돌 방지) |

### 1.3 비동기 통신 모델

```
  앱 → 데몬: 비동기 (요청을 쏟아붓는다)
  데몬 → 아두이노: 동기식 (큐에서 하나씩 꺼내 처리)

  App A ──[요청, 요청, 요청]──┐
                              │
  App B ──[요청, 요청]────────┤  ┌──────────────┐               ┌─────────┐
                              ├─→│  데몬        │──[하나씩]────→│ Arduino │
  App C ──[요청]──────────────┤  │  ┌────────┐  │←─[응답]───────│         │
                              │  │  │스케줄링│  │               └─────────┘
                              │  │  │  큐    │  │
                              │  │  └────────┘  │
                              │  └──────────────┘
                              │
                ←── 비동기 ──→                   ←─── 동기 ───→
```

**왜 비동기인가**: 앱 → 데몬이 동기식이면 각 앱이 응답을 받기 전까지
다음 요청을 보낼 수 없다. 그러면 데몬의 큐에는 항상 요청이 1개뿐이고,
스케줄링 정책(FIFO, 우선순위, EDF 등)을 비교하는 것이 무의미해진다.
비동기로 해야 큐에 요청이 쌓이고, 정책 간 차이가 드러난다.

**응답 매칭**: 앱이 요청을 여러 개 보내므로, 응답이 어떤 요청에 대한 것인지
REQ_ID로 매칭한다. UART의 SEQ와 같은 역할이지만, 다중 앱 환경이므로
4B로 확장했다.

## 2. 소켓 설정

```
  소켓 타입:  SOCK_STREAM (TCP-like, 연결 지향)
  소켓 경로:  /tmp/io_scheduler.sock
  최대 연결:  backlog = 16
```

SOCK_STREAM을 사용하는 이유: 순서 보장과 흐름 제어를 커널에 맡긴다.
SOCK_DGRAM은 메시지 경계는 보존하지만, 대량 요청 시 커널 버퍼 초과로
메시지가 유실될 수 있다.

## 3. 메시지 타입

모든 메시지는 TYPE 필드로 종류를 구분한다.

| 코드 | 이름 | 방향 | 설명 |
|------|------|------|------|
| `0x01` | `CONNECT` | 앱 → 데몬 | 세션 시작, 앱 이름 등록 |
| `0x02` | `CONNECT_ACK` | 데몬 → 앱 | 연결 승인, CLIENT_ID 부여 |
| `0x03` | `DISCONNECT` | 앱 → 데몬 | 세션 종료 |
| `0x10` | `IO_REQUEST` | 앱 → 데몬 | I/O 요청 (스케줄링 힌트 포함) |
| `0x11` | `IO_RESPONSE` | 데몬 → 앱 | I/O 응답 (결과 데이터) |
| `0x20` | `GET_STATUS` | 앱 → 데몬 | 데몬 상태 조회 |
| `0x21` | `STATUS_REPORT` | 데몬 → 앱 | 상태 조회 응답 |

## 4. 스케줄링 정책 설정

스케줄링 정책은 **데몬 시작 시 결정**되며, 런타임에 변경할 수 없다.
변경하려면 데몬을 재시작해야 한다.

```
  # 커맨드라인 인자로 지정
  ./io_daemon --policy fifo
  ./io_daemon --policy priority
  ./io_daemon --policy sjf
  ./io_daemon --policy edf
  ./io_daemon --policy rr
```

**런타임 변경을 지원하지 않는 이유**:

정책 전환 시점에 큐에 이미 쌓여 있는 요청들을 어떻게 처리할지가
모호하다. FIFO로 쌓인 요청들을 갑자기 우선순위 순으로 재정렬해야 하고,
전환 전후의 공정성 보장도 어렵다. 그리고 여러 앱이 정책 변경 권한을
가지면 서로 다른 정책을 요청하는 충돌이 발생한다.

이 프로젝트의 목적은 정책 간 **성능 비교**이지, 런타임에 매끄럽게
전환하는 것이 아니다. 벤치마크는 정책별로 데몬을 재시작하면서
같은 워크로드를 반복 실행하면 된다.

**정책 코드** (GET_STATUS 응답에서 현재 정책을 조회할 때 사용):

| 코드 | 정책 | 사용하는 힌트 필드 |
|------|------|-------------------|
| `0x01` | FIFO | 없음 (도착 순서) |
| `0x02` | 우선순위 기반 | PRIORITY |
| `0x03` | SJF (Shortest Job First) | 없음 (데몬이 추정) |
| `0x04` | SRTF (Shortest Remaining Time First) | 없음 (데몬이 추정, 큐 레벨에서만 적용 — 디스패치된 요청은 비선점) |
| `0x05` | EDF (Earliest Deadline First) | DEADLINE |
| `0x06` | 라운드 로빈 | 없음 (클라이언트 단위 순환: 클라이언트를 돌아가며 각 큐에서 1개씩 디스패치) |

## 5. 메시지 구조

### 5.1 공통 헤더

모든 메시지는 다음 헤더로 시작한다.

```
       0        1        2        3
  ┌─────────────────┬────────┬────────┐
  │    LEN (2B)     │  TYPE  │  CID   │
  └─────────────────┴────────┴────────┘
```

| 필드 | 크기 | 설명 |
|------|------|------|
| LEN | 2B | 헤더 포함 전체 메시지 길이 (리틀 엔디안) |
| TYPE | 1B | 메시지 타입 코드 |
| CID (CLIENT_ID) | 1B | 데몬이 CONNECT_ACK에서 부여한 클라이언트 식별자. CONNECT 시에는 0x00 |

**LEN 필드가 필요한 이유**: Unix 도메인 소켓(SOCK_STREAM)은 스트림이라
메시지 경계가 없다. 수신 측은 먼저 2B를 읽어 메시지 크기를 알아낸 뒤,
해당 크기만큼 추가로 읽어야 한다.

**SOF/EOF/CRC가 없는 이유**: 커널이 데이터의 순서, 무결성, 전달을
보장하므로 유실이나 노이즈가 발생하지 않는다.


### 5.2 CONNECT (0x01)

앱이 데몬에 최초 접속할 때 보내는 메시지다.

```
       0        1        2        3        4       5       5+N
  ┌─────────────────┬────────┬────────┬────────┬───────···───────┐
  │    LEN (2B)     │  TYPE  │  0x00  │NAME_LEN│    APP_NAME     │
  └─────────────────┴────────┴────────┴────────┴───────···───────┘
```

| 필드 | 오프셋 | 크기 | 설명 |
|------|--------|------|------|
| LEN | 0 | 2B | 전체 메시지 길이 |
| TYPE | 2 | 1B | `0x01` |
| CID | 3 | 1B | `0x00` (아직 미할당) |
| NAME_LEN | 4 | 1B | 앱 이름 길이 (최대 31) |
| APP_NAME | 5 | 가변 | 앱 이름 (ASCII, null 종료 없음) |

**앱 이름을 보내는 이유**: 스케줄링 로그와 벤치마크에서
"어떤 앱이 어떤 요청을 보냈는지" 추적하기 위함이다.
CLIENT_ID만으로는 숫자라서 사람이 읽기 어렵다.


### 5.3 CONNECT_ACK (0x02)

데몬이 연결을 승인하며 CLIENT_ID를 부여한다.

```
       0        1        2        3        4
  ┌─────────────────┬────────┬────────┬────────┐
  │    LEN (2B)     │  TYPE  │  CID   │ STATUS │
  └─────────────────┴────────┴────────┴────────┘
```

| 필드 | 오프셋 | 크기 | 설명 |
|------|--------|------|------|
| LEN | 0 | 2B | 5 (고정) |
| TYPE | 2 | 1B | `0x02` |
| CID | 3 | 1B | 부여된 클라이언트 ID (1~255) |
| STATUS | 4 | 1B | `0x00`=성공, `0xFF`=연결 거부 (슬롯 부족) |


### 5.4 DISCONNECT (0x03)

앱이 세션을 종료할 때 보낸다. 데몬은 해당 앱의 미처리 요청을
큐에서 제거하고 CLIENT_ID를 회수한다.

```
       0        1        2        3
  ┌─────────────────┬────────┬────────┐
  │    LEN (2B)     │  TYPE  │  CID   │
  └─────────────────┴────────┴────────┘
```

| 필드 | 오프셋 | 크기 | 설명 |
|------|--------|------|------|
| LEN | 0 | 2B | 4 (고정) |
| TYPE | 2 | 1B | `0x03` |
| CID | 3 | 1B | 클라이언트 ID |


### 5.5 IO_REQUEST (0x10)

I/O 요청 메시지다. 스케줄링 힌트를 포함한다.

```
       0        1        2        3        4        5        6        7
  ┌─────────────────┬────────┬────────┬───────────────────────────────────┐
  │    LEN (2B)     │  TYPE  │  CID   │              REQ_ID (4B)          │
  ├────────┬────────┼────────┼────────┴─────────────────┬─────────────────┤
  │  CMD   │  DEV   │  PRI   │       DEADLINE (4B)      │  PAYL_LEN (2B)  │
  ├────────┴────────┴────────┴──────────────────────────┴─────────────────┤
  │                          PAYLOAD (가변)                               │
  └───────────────────────────────────────────────────────────────────────┘
       8        9       10       11       12       13       14       15
```

| 필드 | 오프셋 | 크기 | 설명 |
|------|--------|------|------|
| LEN | 0 | 2B | 전체 메시지 길이 |
| TYPE | 2 | 1B | `0x10` |
| CID | 3 | 1B | 클라이언트 ID |
| REQ_ID | 4 | 4B | 요청 ID (앱이 생성, 리틀 엔디안). 데몬은 (CID, REQ_ID) 쌍으로 응답을 라우팅한다 |
| CMD | 8 | 1B | 명령 코드 |
| DEV | 9 | 1B | 대상 디바이스 ID |
| PRI | 10 | 1B | 우선순위 (0=최고, 255=최저) |
| DEADLINE | 11 | 4B | 데드라인 (ms 단위, 요청 시점 기준 상대 시간. 0=제한 없음) |
| PAYL_LEN | 15 | 2B | 페이로드 길이 |
| PAYLOAD | 17 | 가변 | 제어 데이터 (WRITE 시 사용) |

**CMD 코드** (UART 프로토콜과 동일):

| 코드 | 이름 | 설명 |
|------|------|------|
| `0x01` | `READ` | 센서 값 읽기 |
| `0x02` | `WRITE` | 액추에이터 제어 |
| `0x03` | `PING` | 연결 확인 |
| `0x04` | `INFO` | 디바이스 정보 조회 |

**스케줄링 힌트 필드 설명**:

```
  PRI (1B):
    0       = 최고 우선순위 (긴급 제어, 비상 정지 등)
    1~63    = 높음
    64~191  = 보통 (기본값: 128)
    192~255 = 낮음 (백그라운드 폴링 등)

  DEADLINE (4B, ms):
    요청 제출 시점으로부터의 상대 시간.
    0 = 데드라인 없음 (무제한 대기 가능)

    예: DEADLINE = 500
    → "이 요청을 500ms 안에 처리해주세요"
    → EDF 정책에서 마감이 가까운 요청부터 처리
    → 데드라인 초과 시 STATUS = ERR_DEADLINE (0x03) 반환
```

**요청마다 힌트를 지정하는 이유**: 같은 앱이라도 요청의 긴급도가 다르다.
온도 센서 1초 폴링은 PRI=200, DEADLINE=0으로 느긋하게,
비상 정지 명령은 PRI=0, DEADLINE=50으로 긴급하게 보낼 수 있다.
앱 등록 시 한 번만 지정하면 이런 구분이 불가능하다.


### 5.6 IO_RESPONSE (0x11)

I/O 요청에 대한 응답이다.

```
       0        1        2        3        4        5        6        7
  ┌─────────────────┬────────┬────────┬───────────────────────────────────┐
  │    LEN (2B)     │  TYPE  │  CID   │              REQ_ID (4B)          │
  ├────────┬────────┴────────┼────────┴───────────────────────────────────┤
  │ STATUS │ PAYL_LEN (2B)   │              PAYLOAD (가변)                │
  └────────┴─────────────────┴────────────────────────────────────────────┘
       8        9       10       11
```

| 필드 | 오프셋 | 크기 | 설명 |
|------|--------|------|------|
| LEN | 0 | 2B | 전체 메시지 길이 |
| TYPE | 2 | 1B | `0x11` |
| CID | 3 | 1B | 클라이언트 ID 에코 |
| REQ_ID | 4 | 4B | 요청 ID 에코 |
| STATUS | 8 | 1B | 결과 코드 |
| PAYL_LEN | 9 | 2B | 페이로드 길이 |
| PAYLOAD | 11 | 가변 | 응답 데이터 (센서 값 등) |

**STATUS 코드**:

| 코드 | 이름 | 설명 |
|------|------|------|
| `0x00` | `OK` | 성공 |
| `0x01` | `ERR_UNKNOWN_DEV` | 존재하지 않는 디바이스 |
| `0x02` | `ERR_UNKNOWN_CMD` | 지원하지 않는 명령 |
| `0x03` | `ERR_DEADLINE` | 데드라인 초과 (큐에서 대기 중 만료) |
| `0x04` | `ERR_DEV_TIMEOUT` | 아두이노 디바이스 응답 없음 |
| `0x05` | `ERR_QUEUE_FULL` | 데몬 큐 포화 |
| `0x06` | `ERR_INVALID_CLIENT` | 유효하지 않은 CLIENT_ID |


### 5.7 GET_STATUS (0x20) / STATUS_REPORT (0x21)

데몬의 현재 상태를 조회한다. 벤치마크와 디버깅에 사용한다.

**요청 (GET_STATUS)**:

```
       0        1        2        3
  ┌────────┬────────┬────────┬────────┐
  │    LEN (2B)     │  TYPE  │  CID   │
  └────────┴────────┴────────┴────────┘
```

**응답 (STATUS_REPORT)**:

```
       0        1        2        3        4        5
  ┌─────────────────┬────────┬────────┬────────┬────────┐
  │    LEN (2B)     │  TYPE  │  CID   │  POL   │ Q_LEN  │
  ├─────────────────┴────────┴────────┼────────┼────────┤
  │            TOTAL_REQS (4B)        │ C_CNT  │AVG_WAIT│
  ├────────┬──────────────────────────┴────────┴────────┘
  │AVG_WAIT│
  └────────┘
       6        7        8        9       10       11  12
```

| 필드 | 오프셋 | 크기 | 설명 |
|------|--------|------|------|
| LEN | 0 | 2B | 13 (고정) |
| TYPE | 2 | 1B | `0x21` |
| CID | 3 | 1B | 클라이언트 ID 에코 |
| POL | 4 | 1B | 현재 스케줄링 정책 코드 |
| Q_LEN | 5 | 1B | 현재 큐에 대기 중인 요청 수 |
| TOTAL_REQS | 6 | 4B | 데몬 시작 이후 처리된 총 요청 수 |
| C_CNT | 10 | 1B | 현재 연결된 클라이언트 수 |
| AVG_WAIT | 11 | 2B | 최근 평균 대기 시간 (ms) |

## 6. 통신 흐름

### 6.1 세션 수립

```
  App                              Daemon
   │                                 │
   │──── CONNECT ───────────────────→│  앱 이름: "temp_monitor"
   │                                 │  CLIENT_ID 할당: 0x01
   │←─── CONNECT_ACK ────────────────│  CID=0x01, STATUS=OK
   │                                 │
   │  (이후 모든 메시지에 CID=0x01 사용)
```

### 6.2 I/O 요청 (비동기)

```
  App (CID=0x01)                   Daemon                      Arduino
   │                                 │                            │
   │── IO_REQ (REQ=1, PRI=128) ─────→│                            │
   │── IO_REQ (REQ=2, PRI=0)   ─────→│  큐: [REQ=2, REQ=1]        │
   │── IO_REQ (REQ=3, PRI=200) ─────→│  큐: [REQ=2, REQ=1, 3]     │
   │                                 │                            │
   │                                 │  정책: 우선순위 기반       │
   │                                 │  REQ=2 (PRI=0) 먼저 처리   │
   │                                 │──── UART 요청 ────────────→│
   │                                 │←─── UART 응답 ─────────────│
   │←── IO_RESP (REQ=2, OK) ─────────│                            │
   │                                 │                            │
   │                                 │  REQ=1 (PRI=128) 처리      │
   │                                 │──── UART 요청 ────────────→│
   │                                 │←─── UART 응답 ─────────────│
   │←── IO_RESP (REQ=1, OK) ─────────│                            │
   │                                 │                            │
   │                                 │  REQ=3 (PRI=200) 처리      │
   │                                 │──── UART 요청 ────────────→│
   │                                 │←─── UART 응답 ─────────────│
   │←── IO_RESP (REQ=3, OK) ─────────│                            │
```

### 6.3 다중 클라이언트

```
  App A (CID=1)                    Daemon                      Arduino
  App B (CID=2)                      │                            │

   A── IO_REQ (PRI=100) ────────────→│                            │

   │                                 │                            │
   │                                 │  A의 PRI=10 먼저 처리      │
   │                                 │──── UART ─────────────────→│
   │                                 │←─── UART ──────────────────│
   A←── IO_RESP (OK) ────────────────│                            │
   │                                 │                            │
   │                                 │  B의 PRI=50 처리           │
   │                                 │──── UART ─────────────────→│
   │                                 │←─── UART ──────────────────│
   B←── IO_RESP (OK) ────────────────│                            │
   │                                 │                            │
   │  (이하 생략)                    │                            │
```

```
  App (CID=1)                      Daemon
   │                                 │
   │── IO_REQ (DEADLINE=100) ───────→│  큐에 삽입
   │                                 │

```
  App (CID=1)                      Daemon
   │                                 │
   │──── DISCONNECT ────────────────→│  CID=1의 미처리 요청 제거
   │                                 │  CID=1 슬롯 회수
   │  (소켓 close)                   │
```

### 6.6 비정상 종료 (앱 크래시)

```
  App (CID=1)                      Daemon
   │                                 │
   │  (프로세스 비정상 종료)          │
   │  (커널이 소켓 자동 close)       │
   │                                 │  read() = 0 (EOF 감지)

## 7. 클라이언트 라이브러리 API

소켓 통신의 세부사항을 숨기고, 앱 개발자에게 간결한 인터페이스를 제공한다.

### 7.1 API 목록

| 함수 | 설명 |
|------|------|
| `io_connect(app_name)` | 데몬에 연결, 핸들 반환 |
| `io_submit(handle, cmd, dev, pri, deadline, payload, &req_id)` | 비동기 요청 제출, REQ_ID 반환 |
| `io_wait(handle, req_id, &result, timeout_ms)` | 특정 요청의 응답 대기 |
| `io_read(handle, dev, &value)` | 동기식 읽기 (submit + wait 래퍼) |
| `io_write(handle, dev, value)` | 동기식 쓰기 (submit + wait 래퍼) |
| `io_get_status(handle, &status)` | 데몬 상태 조회 |
| `io_disconnect(handle)` | 세션 종료 |

## 8. 데몬 ↔ UART 매핑

데몬은 IPC 메시지를 수신하면 UART 프레임으로 변환하여 아두이노에 전달한다.

```
  IPC IO_REQUEST                         UART 요청 프레임
  ┌───────────────────────┐             ┌──────────────────────────┐
  │ CMD      ─────────────┼────────────→│ CMD                      │
  │ DEV      ─────────────┼────────────→│ DEV                      │
  │ PAYLOAD  ─────────────┼────────────→│ PAYLOAD (하위 2B)        │
  │                       │             │                          │
  │ REQ_ID   ─── 데몬 내부 매핑 ───────→│ SEQ (데몬이 자체 관리)   │
  │ PRI      ─── 스케줄링에만 사용      │                          │
  │ DEADLINE ─── 스케줄링에만 사용      │                          │
  └───────────────────────┘             └──────────────────────────┘
```

PRI, DEADLINE은 데몬 내부의 스케줄링 큐에서만 사용된다.
아두이노는 스케줄링을 알 필요가 없다. 아두이노에게는 그냥
"이 디바이스에서 이 명령을 실행해"라는 요청만 도착한다.

## 9. 설계 결정 요약

| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| 통신 채널 | Unix 도메인 소켓 | 같은 호스트 내 IPC에 최적. TCP 오버헤드 없음, 파일 권한으로 접근 제어 가능 |
| 소켓 타입 | SOCK_STREAM | 순서 보장 + 흐름 제어를 커널에 위임. DGRAM은 대량 요청 시 유실 위험 |
| 앱→데몬 방식 | 비동기 | 큐에 요청을 쌓아야 스케줄링 정책 비교가 유의미 |
| REQ_ID 크기 | 4B | 다중 앱이 동시에 요청하므로 1B SEQ로는 충돌 가능 |
| 힌트 지정 단위 | 요청마다 | 같은 앱이라도 요청별 긴급도가 다름. 앱 단위 지정은 세밀한 제어 불가 |
| 프레임 크기 | 가변 길이 | 라즈베리파이는 메모리 여유. LEN 필드로 경계 관리 |
| 무결성 검증 | 없음 | 커널이 보장. CRC/체크섬 불필요 |
| 정책 변경 | 데몬 시작 시 고정 | 런타임 변경 시 큐 재정렬 문제, 다중 앱 충돌 문제 발생. 벤치마크는 재시작으로 충분 |
